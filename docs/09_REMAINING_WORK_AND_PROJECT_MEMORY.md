## 1. 지금까지 구상한 게임

### 한 줄 콘셉트

플레이어는 몬스터 측 던전 복구 및 위생 관리업체의 사장이며, 인간 모험가들이 털고 간 던전을 청소, 복구, 은폐, 회수하면서 돈과 평판과 리스크를 관리한다.

### 핵심 재미

이 게임의 중심은 전투가 아니라 현장 운영이다.

플레이어가 계속 고민해야 하는 축:

- 돈을 벌 것인가, 현장 품질을 챙길 것인가
- 마왕성 신뢰를 챙길 것인가, 인간 길드 평판을 챙길 것인가
- 유품/시체/장비를 정식 반환할 것인가, 암시장에 넘길 것인가
- 직원 피로와 부상을 감수할 것인가
- 위험 이벤트를 해결할 것인가, 철수할 것인가

### 장르 방향

현재 기준:

```text
탑다운 사무실 허브
-> 탑다운 던전 현장 작업
-> 위험 이벤트 때만 짧은 SRPG 전투
-> 현장 복귀
-> 정산/보고
-> 다음 날
```

전투는 본체가 아니다.

권장 비중:

```text
경영/운영: 70~80%
현장 작업: 15~25%
SRPG 전투: 5~10%
```

SRPG는 긴 캠페인 전투가 아니라, 현장 사고를 짧고 명확하게 해결하는 이벤트여야 한다.

## 2. 스토리와 세계관

### 기본 세계

인간 모험가들은 던전을 공격하고, 보물을 훔치고, 함정을 부수고, 성수와 피와 점액으로 현장을 더럽힌다.

마왕성이나 던전 주인은 체면과 운영을 위해 빠르게 현장을 복구해야 한다. 그래서 플레이어의 회사가 하청을 받는다.

### 플레이어 회사

회사 역할:

- 던전 청소
- 오염 제거
- 시체/유품 회수
- 함정 재설치
- 침입 흔적 은폐
- 인간 측 반환/거래 처리
- 위험 상황 진압

회사 정체성:

```text
몬스터 사회의 지저분한 하청업체
마왕성 기준으로는 필요하지만 낮게 보는 외주업체
인간 측 기준으로는 수상하지만 유품 반환을 해주는 회색지대 업체
```

### 톤

톤은 어두운 판타지지만, 공포보다 블랙 코미디에 가깝다.

예시:

- “용사 파티가 또 성수병을 깨고 갔습니다.”
- “시체 포대 추가 요금은 별도입니다.”
- “인간 길드 반환 서류와 마왕성 은폐 보고서가 서로 맞지 않습니다.”
- “함정은 고쳤는데, 고친 사람이 밟았습니다.”

### 첫 번째 의뢰 느낌

초기 계약은 튜토리얼 겸 회사 정체성을 보여주는 작업이어야 한다.

예시 구성:

- 슬라임 오염 청소
- 고장난 함정 수리
- 뼈더미/유품 회수
- 현장에 남은 탐사대 잔당 또는 고용 검문관과 충돌

이 의뢰 하나에서 게임의 핵심을 모두 보여줘야 한다.

```text
현장 작업
-> 위험 이벤트
-> 짧은 SRPG 전투
-> 같은 현장으로 복귀
-> 남은 작업 마무리
-> 정산
```

## 3. 지금까지 구현된 것

### 프로젝트 기반

- Godot 4.7 프로젝트
- 메인 씬: `res://scenes/Office.tscn`
- Autoload: `GameState`
- 사무실, 던전, 전투 테스트 씬 존재

### 사무실

현재 사무실에서 가능한 것:

- 사장 캐릭터 직접 조작
- 의뢰 게시판
- 직원 책상
- 장비 선반
- 장부
- 출동문
- 회사 상태 HUD

아직 부족한 것:

- 직원/장비 선택이 실제 시스템적으로 깊게 연결되어 있지 않다.
- 사무실 UI가 프로토타입 느낌이다.
- 하루 진행/정산/다음 날 루프가 아직 약하다.

### 플레이어 캐릭터

구현된 것:

- 사용자가 가져온 주인공 픽셀 아트 적용
- 8방향 idle/walk 애니메이션 사용
- 사무실/던전에서 같은 캐릭터 조작

주의:

- 전투에서는 아직 정적인 남쪽 방향 이미지를 사용한다.
- 전투용 방향/포즈/피격/방어/스킬 애니메이션은 아직 없다.

### 던전 현장

구현된 것:

- 던전 현장 테스트 씬
- 작업 오브젝트 상호작용
- 작업 완료 상태 저장
- 전투 이벤트 발생
- 전투 승리 후 같은 던전 위치로 복귀

중요하게 고친 점:

- 전투 후 시작 위치로 돌아가는 문제가 있었다.
- 지금은 전투 진입 전 `player.position`을 저장하고, 전투 후 같은 현장 위치로 돌아오도록 수정했다.

아직 부족한 것:

- 작업 종류가 적다.
- 작업에 시간, 위험도, 직원 보정이 붙어 있지 않다.
- 던전 현장 목표가 아직 단순하다.

### 전투

현재 구현된 것:

- 6x6 전투 격자
- 보스 1명
- 적 2명
- 행동 메뉴:
  - 이동
  - 공격
  - 방어
  - 스킬
  - 철수
  - 대기
- 이동 선택:
  - 이동 가능 칸 표시
  - 마우스 클릭 이동
  - WASD/방향키 커서 이동
  - Enter/Space 확정
  - Esc/우클릭 취소
- 공격 선택:
  - 인접 적만 대상
  - 대상 후보 표시
  - 커서로 대상 순환
  - 대상 HP/예상 피해 표시
- 스킬 선택:
  - 사거리 2칸
  - 전투당 1회
  - 대상 후보 표시
  - 대상 HP/예상 피해 표시
- 방어:
  - 다음 적 피해 감소
- 철수:
  - 전투 해결 없이 던전으로 복귀
- 대기:
  - 턴 종료
- 적 턴:
  - 인접하면 공격
  - 아니면 보스 쪽으로 접근

아직 부족한 것:

- 전투 목표가 아직 “적 제압” 중심이다.
- 적 위협 범위가 보이지 않는다.
- 지형/장애물/목표 오브젝트가 없다.
- 적 AI가 단순 추적뿐이다.
- 스킬이 실제 테마와 강하게 연결되어 있지 않다.
- 전투 화면은 기능 중심이며 아직 예쁘거나 완성된 UI가 아니다.

## 4. 우리가 개발을 진행하려고 한 방식

### 기본 원칙

큰 시스템을 한 번에 만들지 않는다.

작은 플레이 가능한 slice를 만든다.

각 slice는 다음 기준을 가진다.

```text
1. 화면에서 보인다.
2. 조작할 수 있다.
3. 상태가 바뀐다.
4. 다음 씬/루프와 연결된다.
5. Godot headless 로드로 최소 검증한다.
```

### 현재까지의 진행 순서

진행한 큰 흐름:

```text
메뉴형 경영 프로토타입
-> 탑다운 사무실/던전 방향으로 수정
-> 주인공 직접 조작
-> 의뢰/던전 작업/전투 연결
-> 전투 후 현장 복귀 위치 보존
-> 주인공 픽셀 아트 적용
-> 전투 적 비주얼 교체
-> SRPG 연구 문서 작성
-> SRPG 행동 메뉴/상태 머신 1차 구현
-> 이동/공격/스킬 선택 UX 보정
```

### 앞으로도 유지할 개발 방식

앞으로도 아래 순서가 맞다.

```text
문서 기준 정리
-> 작은 기능 slice
-> headless 로드 검증
-> 가능하면 실제 플레이 감각 확인
-> 커밋/푸시
-> 다음 slice
```

중요:

- 전투 기능을 깊게 파기 전에, 경영 루프와 현장 루프를 계속 연결해야 한다.
- SRPG는 길게 만들면 게임 본체를 잡아먹는다.
- 전투는 “현장 사고 해결 도구”여야 한다.

## 5. 내가 부족했던 점과 교훈

### 1. 처음에 SRPG를 너무 얕게 봤다

문제:

- 격자와 턴만 있으면 SRPG처럼 보일 거라고 처리했다.
- 이동 선택, 행동 선택, 대상 선택, 확정 흐름이 약했다.
- 적 클릭 즉시 공격, 칸 클릭 즉시 이동 같은 보드게임식 임시 조작을 넣었다.

교훈:

- SRPG는 클릭 처리보다 “선택 단계가 보이는 것”이 중요하다.
- 플레이어가 무엇을 선택 중인지, 무엇을 확정하는지 항상 보여야 한다.

### 2. 이동/공격 입력 검증이 부족했다

문제:

- 행동 메뉴를 넣으면서 WASD 이동이 막혔다.
- 보드 클릭도 UI 이벤트에 먹혀 제대로 안 될 수 있는 구조였다.

고친 점:

- 보드 입력 전용 `Control` 레이어를 추가했다.
- WASD/방향키는 전투 커서 이동으로 바꿨다.
- Enter/Space 확정 흐름을 붙였다.

교훈:

- Godot UI와 게임 보드 입력이 섞이면 Control 이벤트 소비를 조심해야 한다.
- 실제 플레이 조작을 기준으로 검증해야 한다.

### 3. 공격/스킬 대상 선택을 제대로 드러내지 못했다

문제:

- 공격/스킬 버튼이 조건 안 맞으면 그냥 비활성이라 플레이어가 왜 안 되는지 보기 어려웠다.
- 공격/스킬 모드에 들어가도 “대상을 고른다”는 느낌이 약했다.

고친 점:

- 공격/스킬 버튼은 누를 수 있게 두고, 대상이 없으면 이유를 표시한다.
- 대상 후보 표시, 커서 후보 순환, 대상 HP/예상 피해를 붙였다.

교훈:

- 비활성 버튼만으로는 학습이 안 된다.
- 조건이 안 될 때도 이유를 보여줘야 한다.

### 4. 문서 언어를 처음에 사용자가 보기 어렵게 만들었다

문제:

- SRPG 리서치 문서를 처음 영어 본문으로 작성했다.

고친 점:

- 문서를 한글 본문으로 다시 작성했다.
- 영어는 이미지 생성 프롬프트처럼 필요한 곳에만 남겼다.

교훈:

- 이 프로젝트의 작업 문서는 기본 한글이어야 한다.
- 영어는 도구 입력용 프롬프트나 원문 인용 링크에만 쓴다.

### 5. 사용자 의도를 늦게 반영한 부분이 있었다

예:

- 전투 승리 후 던전 시작 위치로 돌아가는 문제
- 주인공 이미지 방향 판정
- 8방향 애니메이션 필요
- SRPG 조작 문법

교훈:

- “일단 돌아감”보다 “플레이어가 기대한 게임 문법에 맞는가”를 먼저 봐야 한다.
- 특히 SRPG, 탑다운 이동, 애니메이션 방향 같은 장르 문법은 대충 넘기면 바로 체감된다.

## 6. 지금 남은 일 전체 목록

### P0. 바로 다음에 해야 할 것

#### P0-1. 전투 목표 배너 추가

현재 전투는 아직 적 제압 중심이다.

해야 할 일:

- 전투 상단에 목표 표시
- 예:
  - `탐사대 잔당을 제압하라`
  - `경보를 울리기 전에 검문관을 막아라`
  - `3턴 동안 시체 포대를 보호하라`
- `combat_event` 데이터에서 목표 텍스트를 받을 수 있게 한다.

완료 기준:

- 전투 시작 시 목표가 보인다.
- 승리/패배 메시지가 목표와 연결된다.

#### P0-2. 적 위협 범위 표시

현재 플레이어는 적이 다음 턴에 어디를 때릴지 모른다.

해야 할 일:

- 적 이동 가능 범위 또는 공격 가능 범위 표시
- 최소 버전:
  - 인접 공격 적은 다음 턴 위험 칸을 보라/어두운 빨강으로 표시
- 이동 선택 중 위험 칸을 같이 볼 수 있게 한다.

완료 기준:

- 이동 전에 위험한 칸을 알 수 있다.
- 방어/대기/철수 선택이 더 의미 있어진다.

#### P0-3. 전투 목표를 적 전멸에서 목표형으로 확장

예시 목표:

- 3턴 버티기
- 경보 장치 파괴
- 시체 포대 보호
- 오염 핵 제거
- 탈출 타일 도달

완료 기준:

- 최소 1개 전투가 적 전멸이 아닌 조건으로 끝난다.

#### P0-4. 전투 UI 정리

현재 기능은 들어갔지만 UI는 임시다.

해야 할 일:

- 행동 메뉴 버튼 선택 상태 더 명확히 표시
- 대상 정보 패널 정리
- 로그가 너무 길어지지 않게 정리
- 키 안내를 문장으로 길게 반복하지 않고 짧게 표현

완료 기준:

- 지금 어떤 모드인지 1초 안에 읽힌다.
- 버튼/대상/타일 색이 서로 충돌하지 않는다.

### P1. 경영 루프 복구/강화

#### P1-1. 의뢰 선택 UI 실제화

현재 계약 데이터는 있지만 UI/선택 경험이 약하다.

해야 할 일:

- 의뢰 게시판에서 여러 의뢰 표시
- 보상/위험/작업 종류 표시
- 선택한 의뢰가 던전 생성과 정산에 반영

완료 기준:

- 플레이어가 “오늘 어느 현장을 갈지” 선택한다.
- 선택이 던전 작업/전투/보상에 영향을 준다.

#### P1-2. 직원 편성

직원은 이 게임의 경영 자원이다.

해야 할 일:

- 직원 목록 UI
- 직원 상태:
  - 피로
  - 부상
  - 특화
- 출동 직원 선택
- 현장 작업 보정 연결

완료 기준:

- 그릭/멜타/볼그/피핏 중 일부를 선택해서 현장 효과가 달라진다.

#### P1-3. 장비 선택

장비는 돈/평판/위험 판단을 강화해야 한다.

해야 할 일:

- 장비 목록 UI
- 출동 장비 선택
- 장비 효과를 현장/정산에 연결

예:

- 소독 점액통: 오염 처리 보정
- 사체 운반 포대: 시체 회수 보상 증가
- 검은 장부: 돈 증가, 불법 리스크 증가

완료 기준:

- 장비 선택이 정산 결과를 바꾼다.

#### P1-4. 정산 보고 화면

현재 정산은 결과 텍스트 중심이다.

해야 할 일:

- 별도 결과 보고 화면 또는 사무실 보고 패널
- 오늘 처리한 작업 수 표시
- 보상/평판/리스크 변화 표시
- 전투 결과 반영
- 다음 날 진행 버튼

완료 기준:

- 하루 루프가 명확하게 닫힌다.

### P2. 던전 현장 작업 확장

#### P2-1. 작업 오브젝트 종류 확장

현재 작업이 적다.

추가 후보:

- 시체 포대 끌기
- 유품 줍기
- 오염 핵 청소
- 경보 장치 해제
- 고장난 함정 재설치
- 무너진 잔해 제거
- 인간 장비 감정

완료 기준:

- 같은 던전에서도 작업 순서 선택이 생긴다.

#### P2-2. 작업 시간/위험도

각 작업은 단순 클릭 완료보다 선택을 만들어야 한다.

예:

- 오래 걸리지만 안전하게 처리
- 빠르게 처리하지만 위생/리스크 손해
- 직원 특화로 시간 단축

완료 기준:

- 작업 방식에 따라 정산이 달라진다.

#### P2-3. 현장 내 전투 이벤트 조건화

전투는 특정 행동 때문에 터지는 게 좋다.

예:

- 경보 장치 건드림
- 시체 포대 회수 시 인간 잔당 등장
- 오염 핵 청소 중 몬스터 변이 발생

완료 기준:

- 전투가 랜덤 팝업이 아니라 현장 행동의 결과로 느껴진다.

## 7. 다음 작업 추천 순서

가장 추천하는 다음 5개:

```text
1. 직원/장비 편성 UX 다듬기
2. 현장 작업에 직원/장비별 즉시 피드백 추가
3. 첫 의뢰 하나를 더 깊게 설계
4. 정산 보고 화면 시각 정리
5. 과거 Main.gd 메뉴형 프로토타입 archive 여부 결정
```

이유:

- SRPG는 현재 프로토타입 수준에서 잠시 보류하기로 했다.
- 지금은 플레이어가 사무실에서 의뢰, 직원, 장비를 고르고 현장 결과로 정산을 받는 루프를 굵게 만드는 단계다.
- 다음은 숫자 보정이 실제 플레이 감각으로 보이게 만드는 작업이 좋다.

추천하지 않는 다음 작업:

- 복잡한 전투 스킬트리
- 다인 파티 전투
- 랜덤 던전 생성
- 대형 아트 교체
- UI 전체 리디자인

아직 본체 루프가 약해서, 이런 작업은 지금 하면 범위가 커진다.

## 9. 현재 주의 사항

- `.omx/` 런타임 파일은 계속 dirty일 수 있다. 커밋하지 않는다.
- Godot headless 실행 시 `project.godot`에 `window/size/mode=3`이 자동으로 들어갈 수 있다. 작업 범위와 무관하면 제거한다.
- 문서는 기본 한글로 작성한다.
- 실제 이미지 생성은 별도 요청 전까지 하지 않는다.
- 전투는 깊어질수록 게임 본체를 잡아먹으므로 경영/현장 루프와 계속 연결한다.

## 10. 현재 남은 가장 큰 질문

아직 결정이 필요한 질문:

1. 첫 MVP는 며칠짜리 루프로 볼 것인가?
   - 3일
   - 5일
   - 7일

2. 첫 정식 의뢰는 하나를 깊게 만들 것인가, 3개를 얕게 만들 것인가?

3. 전투 실패는 어떤 손해인가?
   - 보너스 없음
   - 직원 부상
   - 현장 작업 일부 실패
   - 마왕성 신뢰 하락

4. 철수는 언제 허용할 것인가?
   - 항상 허용
   - 특정 목표에서는 금지
   - 철수 시 정산 패널티

5. 직원은 전투에 직접 유닛으로 나올 것인가, 스킬/지원 호출로만 나올 것인가?

현재 추천 답:

```text
MVP는 5일 루프.
첫 의뢰 하나를 깊게 만든다.
전투 실패는 보너스 없음 + 약한 신뢰/평판 손실.
철수는 항상 허용하되 정산 패널티.
직원은 처음에는 전투 유닛이 아니라 스킬/지원 호출로만 쓴다.
```

## 11. 한 문장 결론

지금 프로젝트는 “직접 조작하는 던전 청소 회사 경영 게임”으로 방향이 잡혔고, SRPG 전투는 이제 조작 초안이 생겼다. 다음 핵심은 전투를 목표형 사고 해결로 만들고, 다시 의뢰/직원/장비/정산 경영 루프를 굵게 연결하는 것이다.
